# 策略广场一期业务需求与职责分配

版本：v0\.9 业务需求职责分配草案

日期：2026\-05\-27

范围：基于当前前端原型拆解，一期覆盖 Strategy 下的 Marketplace、Activity、Deployed 三个 Tab，包含个性化策略推荐、策略详情手动回测和策略消息推送。Studio、Markets、Portfolio、News、Brokers 等其他模块暂不纳入。

## 一期目标

策略广场一期目标是让用户可以完成从“发现策略”到“收藏/观察策略”再到“部署并查看执行结果”的基础闭环：

1. 在 Marketplace 浏览后端维护的策略池、个性化推荐策略、排行榜、策略详情和历史指标。

2. 将感兴趣的策略加入 Activity，形成用户自己的策略观察列表，并持续查看观察期收益、策略状态、信号和消息。

3. 对已加入的策略，在策略详情页输入时间周期发起手动回测，查看该区间的策略回测结果。

4. 将策略部署到 Paper Trading 或实际券商账户，进入 Deployed 查看账户、持仓、收益、交易明细和策略管理动作。

5. 对收藏或部署的策略接收 Quant Push。推送事件由策略执行侧产生，后端转化为用户消息并调用 Push 服务，Push 服务通过 WebSocket 触达前端。

一期需要支持个性化推荐接入。推荐只负责 Marketplace 中的用户推荐策略展示：策略广场服务每天将可推荐策略同步给推荐服务，推荐服务根据用户返回推荐策略 ID，策略广场后端补齐策略展示信息后返回前端。Marketplace 通用策略池展示、筛选排序、排行榜排序、Push、交易、风控均不需要推荐参与。

## 团队职责边界

### 4\.1 算法开发：策略引擎服务 @杨文园

负责策略从维护到部署运行的生命周期能力：

1. 策略维护：策略定义、策略状态、策略运行参数、策略上下架前的策略侧校验；一期策略默认使用最新定义，不设计版本概念。

2. 用户策略部署：创建部署实例、运行容器、调度状态、停止/恢复、失败重试、健康检查。

3. 用户已部署策略查询：按 user\_id 输出用户已部署策略 ID、deployment\_id、策略关联信息和必要状态，供后端聚合 Activity 部署标记。

4. 策略运行状态同步：向后端提供策略运行状态、部署状态变化和必要事实事件。

5. 部署策略自动化交易：接收执行引擎信号，并与交易侧对接自动交易指令或执行计划。

6. Stop \&amp; Close 协同：配合交易侧完成策略停止后的运行状态同步，让 Activity 可以回到 watching 观察态。

### 4\.2 算法开发：算法策略执行引擎 @丁宇杰

负责策略信号、回测和指标计算能力：

1. 生产策略信号：按策略参数、行情、事件和调度生成买卖、调仓、风控等信号。

2. 策略回测：计算历史收益、收益曲线、交易列表、Sharpe、最大回撤、胜率等指标。

3. 自定义日期区间回测：支持策略详情页手动回测，按用户输入日期区间返回标准回测结果。

4. Activity 观察收益：根据后端传入的收藏开始时间和结束时间计算区间收益率、曲线和关键指标。

5. 策略指标标准化：为 Marketplace、Activity 和 Deployed 对照指标提供统一口径。

6. Push 事实事件：配合策略引擎把策略信号、调仓、风控、里程碑、部署状态等策略执行事件推送给后端。

7. 推送规则调研：本周调研存量策略运行情况，给出一期可推送事件、阈值、频控、文案和严重程度建议。

### 4\.3 后端业务开发：策略广场服务 @王正强

负责策略广场业务数据、用户交互和消息体系：

1. Marketplace 首页展示：策略池列表、个性化推荐策略结果补全、卡片、详情、筛选、排序和榜单展示。

2. 策略展示信息管理：提供策略展示信息维护能力，一期先支持 Excel 导入初始化策略和批量维护展示字段。

3. 策略展示数据存储：存储策略展示信息、收益信息、指标快照、曲线快照、标签、上下架状态和可推荐状态。

4. 推荐服务对接：每天定时将可推荐策略同步给推荐服务，调用推荐接口按用户获取推荐策略 ID，并对返回结果做可展示过滤和详情补全。

5. 用户 Activity：保存用户加入 Activity 的策略、来源、状态、排序、收藏开始时间、移除状态、推送订阅关系。

6. Activity 观察收益：记录用户最新一次加入 Activity 的开始时间，调用策略侧计算区间收益，后端保存快照并返回前端。

7. Activity 部署标记：统一聚合收藏策略和已部署策略信息，返回已部署/未部署、部署状态和入口所需 ID；不返回账户收益、账户号、持仓、订单或成交。

8. 下架策略处理：不自动删除用户已收藏/已部署策略，只返回“已下架”标记；已部署策略继续由交易侧运行到用户主动停止。

9. Quant Push：接收策略执行侧事件，负责消息规则、订阅、幂等、消息落库、已读、点击、调用 Push 服务和触达状态记录。

10. 权限和审计：确保用户只能查看自己的收藏、Activity 和 Push 消息；Deployed、账户和交易权限由交易后端负责。

### 4\.4 交易开发：Deployed 和交易执行 @丁骏\(Jun Ding\)

负责 Deployed 模块和实际交易执行链路。Deployed 相关账户连接、账户收益、持仓、交易明细、部署管理接口由交易后端直接面向前端输出，不经过策略广场业务后端中转：

1. 券商连接：账户授权、账户识别、连接状态、资金和持仓查询。

2. 策略实际执行：接收策略执行计划，完成下单、撤单、成交回报和异常处理。

3. 账户策略交易明细：订单、成交、费用、状态、时间、策略归因、账户归因。

4. 持仓标的：按账户、标的、策略展示持仓、成本、现价、未实现盈亏。

5. 部署策略维度收益：按部署实例计算收益、今日收益、累计收益、回撤、交易次数等。

6. Deployed 前端接口：直接向前端提供已部署策略、账户、收益、持仓、交易明细、策略管理等接口。

|---|---|
|Activity 定位|Activity 是观察工作台，只展示观察、信号、状态、消息和部署标记，不展示账户连接、真实账户收益、真实持仓和交易明细。|

## 待确认细节和本周待办

### 7\.1 推送规则

以下为算法本周调研后需要落定的推送规则细节：

1. 策略执行侧一期需要向后端推送哪些事件类型。

2. watch\-only、paper、live 不同状态下分别推送哪些消息。

3. 盈亏类推送阈值如何定义，例如单日收益、累计收益、回撤、连续亏损、触发止损。

4. 是否需要免打扰时段、频控、去重、合并推送。

5. 用户是否可以按策略或消息类型关闭推送。

6. 策略执行侧推送给后端的事件字段、幂等键、重试机制和去重规则。

7. 推送文案是否需要强制展示风险提示或免责声明。

### 7\.2 Push 服务协议

以下只保留 Push 服务协议待确认细节：

1. 正式环境域名和测试环境域名。

2. 鉴权方式、调用方标识、限流规则。

3. 请求字段必填项、默认值、字段长度限制。

4. 错误码、失败重试、幂等键和投递回执。

5. 前端 WebSocket 的连接地址、鉴权方式、消息结构、重连策略。

6. `priority`、`endTime`、`invalidTime`、`duration`、`comefrom`、`logo`、`miniIcon` 等字段的平台默认配置。

### 7\.3 Activity 和 Deployed 状态同步

以下只保留状态同步实现细节，业务边界已在第 5 章定义：

1. Stop \&amp; Close 执行完成后，交易侧、策略引擎侧或后端哪一方作为状态变化事实来源。

2. Stop \&amp; Close 失败、部分平仓、券商异常时，Activity 是否保持已部署标记。

3. Deployed 停止策略但未平仓的场景是否存在，如存在前端如何区分。

### 7\.4 推荐方案

以下只保留推荐接口和配置落地细节，推荐范围、同步频率和兜底方向已在第 5 章定义：

1. 每日同步给推荐服务的完整字段清单和字段含义。

2. 推荐接口请求路径、请求参数、返回参数、错误码和超时策略。

3. 世界杯期间推荐兜底策略集合由哪一方维护，以及配置生效和刷新方式。

4. 世界杯后的默认兜底配置由推荐侧提供还是后端配置中心维护。

### 7\.5 策略展示数据维护

以下只保留 Excel 导入的执行细节，展示数据来源和维护口径已在第 5 章定义：

1. 基于前端 Marketplace、策略详情、Activity 卡片等页面需要展示的信息，整理 Excel 导入模板字段清单。

2. 是否支持导入预览、错误报告、更新、删除、上下架。

3. Excel 导入失败后的错误提示、部分成功处理和重复策略覆盖规则。

4. Excel 维护字段和算法实时指标字段的冲突处理优先级。

### 7\.6 手动回测

手动回测已确认实时返回给用户查看，不落库。以下只保留接口和交互细节：

1. 日期范围限制，例如最早可选日期、最长区间、是否允许未来日期。

2. 是否限制用户并发次数、每日次数或策略级频率。

3. 回测结果字段，包括收益曲线、指标、交易列表、错误信息和计算耗时。

4. 已下架但已收藏策略是否允许手动回测。

### 7\.7 券商和交易范围

1. 一期真实券商/API 的 P0 范围需要确认。原型拆解图提到 Polymarket、Hyperliquid、Paper Trading，原型数据里还出现 IBKR、Alpaca，需要确认哪些是一期真实支持，哪些只是演示。

2. 是否支持只有 Paper Trading、不支持真实资金交易的灰度形态。

3. 不同券商的币种、交易时间、费用、成交状态如何统一展示。

4. 用户平仓操作是否必须二次确认，是否支持一键 Stop \&amp; Close all positions。

5. 券商连接失败、权限过期、下单失败、部分成交等异常如何展示和推送。
